# Session Extract — 2026-04-13

## Architecture Decisions

### ADR: Inline Discard Banner over window.confirm
**Decision:** Replaced `window.confirm()` in AddModal's tryClose with a `confirmDiscard` state boolean and inline red banner with "Keep editing" / "Discard" buttons.
**Why:** `window.confirm()` blocks the main thread, can't be styled, and behaves unpredictably in the artifact sandbox. Inline banners are consistent, accessible (role="alert"), and auto-dismiss if the user continues editing.
**Tradeoff:** Adds one state variable and 15 lines of JSX per confirmation flow. Acceptable at this scale.
**Rejected:** Custom modal component — overkill for a single confirmation with two buttons.

### ADR: Memory Edits for Behavioral Rules vs Settings > Profile
**Decision:** Universal behavioral rules (apply to every conversation) go in Settings > Profile. Project-scoped facts, session state, and environment constraints go in memory_user_edits.
**Why:** Both land in Claude's context window, but they serve different purposes. Preferences are identity-level ("always do X"). Memories are situational ("project Y is in state Z"). Mixing them degrades signal quality in both systems.
**Tradeoff:** Writing to Settings > Profile requires browser automation (preference-writer skill) since there's no API. Slower than memory_user_edits but correct placement.
**Rejected:** Storing behavioral rules as memory edits — functional but conceptually wrong and bloats the memory cap.

## Reusable Patterns

### Pattern: Dual-Version Artifact Sync Check
- **Problem:** Two files must stay identical except for one known diff point. Drift bugs are silent.
- **Implementation:** After any edit to either file, run `diff App.artifact.jsx App.jsx`. Output must show ONLY the storage layer lines. Anything else is a sync bug.
- **Origin:** JotFolio maintains App.artifact.jsx (window.storage) and App.jsx (localStorage). Both were edited this session; diff confirmed sync.

### Pattern: Quality Gate with Drop Log
- **Problem:** Extraction/summarization skills default to producing something for every section, padding output to justify their existence.
- **Implementation:** Before producing any output, run every candidate through explicit drop criteria. Show the user what was dropped and why. If everything drops, say so. Gives the user override power without requiring Claude to guess.
- **Origin:** Built into the session-extract skill this session.

### Pattern: Browser Automation for Server-Side Settings
- **Problem:** Some settings (Claude user preferences) have no API and no local file. Only accessible through UI.
- **Implementation:** Use Claude in Chrome to navigate to settings URL, read existing content, append new content, save, verify, close. Critical constraint: `form_input` REPLACES the field, it does not append. Must read-then-write-full-content.
- **Origin:** preference-writer skill. Nearly destroyed Gavin's entire preferences XML by writing without reading first.
